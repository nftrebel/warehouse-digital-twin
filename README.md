# 🏭 Цифровой двойник бизнес-процесса складской логистики

> Событийная система реконструкции состояния партий и заказов на основе входящих логов процесса

Веб-приложение на Django, которое принимает события складского процесса через REST API, ведёт неизменяемый журнал событий, восстанавливает текущее состояние объектов и визуализирует ход процесса в режиме реального времени.

---

## 📋 О проекте

Цифровой двойник в данном проекте — это **программная событийно-статусная модель**, которая:

- принимает логи из внешней системы через HTTP POST API;
- сохраняет каждое событие как неизменяемую запись в журнале;
- пересчитывает текущее состояние партий, заказов и складских ячеек;
- отображает реконструированную картину процесса на экранах веб-интерфейса;
- рассчитывает KPI.

**Предметная область** — сквозной процесс складской логистики: приёмка → размещение → хранение → резервирование → комплектация → отгрузка.

## 🚀 Быстрый старт

### 1. Клонирование репозитория

```bash
git clone https://github.com/<your-username>/warehouse-digital-twin.git
cd warehouse-digital-twin
```

### 2. Создание виртуального окружения

```bash
python3 -m venv venv
source venv/bin/activate  # macOS / Linux
```

### 3. Установка зависимостей

```bash
pip install -r requirements.txt
```

### 4. Инициализация базы данных

```bash
python manage.py makemigrations references inventory orders events accounts
python manage.py migrate
```

### 5. Создание демо-пользователей

```bash
python manage.py create_demo_users
```

Будут созданы:
- **admin** / `admin123` — администратор
- **analyst** / `analyst123` — аналитик

### 6. Загрузка демонстрационных данных

```bash
python manage.py seed_demo_data
```

### 7. Запуск сервера

```bash
python manage.py runserver
```

Откройте в браузере: **http://127.0.0.1:8000/**

## 📦 Модель данных

### Сущности

| Сущность | Назначение |
|----------|-----------|
| **Product** | Справочник товарных позиций (SKU, наименование, ед. измерения) |
| **StorageLocation** | Складские локации (приёмка, хранение, комплектация, буфер, отгрузка) |
| **Batch** | Партия товара с встроенным текущим состоянием |
| **CustomerOrder** | Заказ на отгрузку с встроенным состоянием |
| **OrderLine** | Строка заказа (товарная позиция) |
| **BatchReservation** | Резервирование партии под строку заказа (связь M:M) |
| **ProcessEvent** | Неизменяемое событие складского процесса |
| **User** | Учётная запись с ролью (аналитик / администратор) |

### Жизненный цикл партии

```
expected → received → placed → stored → reserved → shipped
                                  ↑                    │
                                  └────────────────────┘
                                  (возврат при частичной отгрузке)
```

При отгрузке заказа связанные партии:
- возвращаются в **stored**, если в них остался товар;
- переходят в **shipped**, если товар закончился;
- остаются в **reserved**, если есть другие активные резервы.

### Жизненный цикл заказа

```
created → picking → assembled → shipped
```

## 🔌 REST API

### Основной эндпоинт приёма событий

```
POST /api/v1/events/
```

Пример запроса:

```json
{
  "event_id": "0f8fad5b-d9cb-469f-a165-70867728950e",
  "event_type": "batch.placed",
  "occurred_at": "2026-03-20T10:15:00Z",
  "source_system": "simulator",
  "warehouse_code": "WH-01",
  "object_type": "batch",
  "object_id": "BATCH-00045",
  "payload": {
    "batch_code": "BATCH-00045",
    "product_sku": "SKU-1001",
    "qty": 120,
    "to_location": "A-01-03"
  }
}
```

### Поддерживаемые типы событий

| Тип | Описание |
|-----|----------|
| `batch.received` | Партия принята на склад |
| `batch.placed` | Партия размещена в ячейке |
| `batch.moved` | Партия перемещена между ячейками |
| `batch.reserved` | Количество зарезервировано под заказ |
| `order.created` | Создан заказ на отгрузку |
| `order.picking_started` | Начата комплектация заказа |
| `order.item_picked` | Подобрана позиция заказа |
| `order.assembled` | Комплектация завершена |
| `shipment.dispatched` | Заказ отгружен |

### Все эндпоинты

| Метод | URL | Назначение |
|-------|-----|-----------|
| `POST` | `/api/v1/events/` | Приём одного события |
| `POST` | `/api/v1/events/bulk/` | Пакетная загрузка событий |
| `GET/POST` | `/api/v1/reference/products/` | Справочник товаров |
| `GET/POST` | `/api/v1/reference/locations/` | Справочник локаций |
| `GET/POST` | `/api/v1/batches/` | Партии |
| `GET/POST` | `/api/v1/orders/` | Заказы |

## 📊 Аналитика и KPI

Экран KPI поддерживает фильтрацию по временному диапазону с быстрыми кнопками (7 / 30 / 90 дней).

**Рассчитываемые метрики:**
- Среднее время приёмка → размещение
- Среднее время комплектация → отгрузка
- Количество отгруженных заказов за период
- Количество просроченных заказов
- Распределение событий по типам и источникам
- Длительности переходов между этапами (для партий и заказов)

### Экспорт отчётов

Доступен экспорт в двух форматах:

| Формат | Содержание |
|--------|-----------|
| **Word (.docx)** | Форматированный документ с таблицами, отдельными страницами для отгруженных и просроченных заказов |
| **Excel (.xlsx)** | 8 листов: KPI, партии по этапам, заказы по этапам, события по типам, по источникам, длительности, отгруженные заказы, просроченные заказы |

Оба формата учитывают выбранный диапазон дат и включают детализацию позиций по каждому заказу.

Команда `seed_demo_data` создаёт полный набор данных, покрывающий все сценарии:

| Задержки и аномалии | 2 просроченных заказа, партия зависшая на приёмке > 48ч |

**Итого:** 12 товаров, 15 локаций, ~15 партий, ~12 заказов, 55+ событий.
